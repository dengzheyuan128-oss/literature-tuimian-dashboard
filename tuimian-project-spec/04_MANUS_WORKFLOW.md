# Manus工作流、提交与反馈规范

> 状态：强制执行  
> 最后更新：2026-01-17

## 核心规则（最高优先级）

> **Manus每完成一次可感知的更新任务，必须提交到GitHub，并形成可审阅的记录。**

**未提交的修改 = 不存在**

---

## 标准工作流程

### Step 1: 任务开始前（轻量规划）

在开始任何实质性修改前，Manus必须先明确：

#### 1.1 数据来源类型

- 来源A：你提供的Excel（院校名录主入口，优先级最高）
- 来源B：你人工确认后补充的单条链接
- ❌ 禁止来源：Manus自主大规模搜索官网

#### 1.2 任务类型

- A｜院校名录数据更新
- B｜功能/页面框架搭建
- C｜脚本/校验规则调整

#### 1.3 任务边界

- 涉及哪些文件
- 明确**不做什么**

> 允许一句话说明，不要求长文档。

---

### Step 2: 执行中（受限执行）

#### 2.1 执行约束

- ❌ 不额外扩展需求
- ❌ 不"顺手优化"无关模块
- ❌ 不引入新依赖

#### 2.2 超出范围的处理

任何超过原任务范围的想法：
1. **暂停执行**
2. 向你反馈
3. 等待确认

---

### Step 3: 完成后（强制GitHub提交）

#### 3.1 提交要求

- **必须提交到GitHub**
- 单一目的提交（不混合多类修改）
- 使用标准Commit Message模板

#### 3.2 Commit Message 模板（强制）

```text
[type] 简要说明本次修改

- 变更范围：
- 是否影响既有结构：是/否
- 是否需要人工复核：是/否
```

**type取值限定**：
- `data`：院校名录/链接更新
- `feat`：新增页面/功能骨架
- `fix`：规则修正/Bug
- `docs`：文档说明

#### 3.3 提交示例

```bash
git add universities.json
git commit -m "[data] 更新第一梯队6所院校链接等级为A

- 变更范围：universities.json（北大、人大、复旦、南大、川大、北师大）
- 是否影响既有结构：否
- 是否需要人工复核：否（已人工验证）"
```

---

### Step 4: 提交后反馈（让你可感知）

提交完成后，Manus必须同步反馈：

1. **GitHub Commit链接**
2. **本次修改的"你需要关注的点"**（1-3条）

反馈格式：
```markdown
## 提交完成 ✅

**Commit**: [链接]

**关注点**:
1. 更新了XX所院校
2. XX所标记为pending_manual
3. 数据质量指标变化：XX → XX
```

---

## PR与直接提交的使用规则

### 允许直接提交的情况

- 数据更新（你已提供链接）
- 纯页面骨架搭建
- 文档补充

### 必须走PR的情况

- 结构调整（schema/API/loader）
- 校验逻辑修改
- 影响多模块的改动

---

## 低打扰原则

为避免你被频繁打断：

- **小改动**：提交即可，不必等待确认
- **中等改动**：提交 + 简要说明
- **大改动**：先方案，等你确认后再动

---

## 违反规则的处理

如出现以下情况：
- 未提交GitHub的修改
- 多次commit无说明
- 未经确认的需求扩展

→ **该修改视为无效，可直接回滚，不进入正式版本**

---

## 文档优先级

在开始任何任务前，Manus必须：

1. 明确当前任务对应哪一份文档
2. 只按该文档中已写明的规则执行
3. 不自行"综合多份文档做推断"

### 权威等级（从高到低）

1. **项目文档（tuimian-project-spec/）** ← 唯一权威
2. GitHub当前代码状态
3. 对话中的补充说明

> 若三者存在冲突，**必须以项目文档为准**。

---

## 发现文档缺失或冲突时

1. **暂停执行**
2. 提交Issue/说明给你
3. 等待文档更新
